# 感じたこと
## 走査用の変数の命名が難しい
Listを走査する時に使用する変数の名前に安直なcurrent_nodeやnext_nodeというものを使用したが、current、nextの意味するところが読み手に簡単に通じるのか自信がなかった。
これは競技プログラミングを行っていた時から思ってはいたことであるが、どこまで通じるのかよくわかっていない。

## dataclassが自分の型を参照できないことがもどかしい
from __future__ import annotationsを使用すれば直接自分の型を使えるらしいが、LeetCodeでの実行時にはこれを使うとエラーになってしまうので、
文字列を使って自分自身を参照する必要があった。文字列だとIDEの恩恵を十分に受けられるのか怪しいため、もどかしく感じた。
が、そもそもleetcode的にここの型を自分で定義することは考えられていなさそうであったのでstep2からはデフォルトで書いてあったものを使用した。

## returnを複数箇所に書くか、１箇所にまとめるかで迷った
たまに聞く話やlinterのルールには、returnを複数箇所に書かないほうがよいということであるが、今回くらいのケースであれば、返せる段階で
さっさと返してしまったほうが余計な変数が減ってむしろ読みやすいのではないかと思ったが、迷った。
